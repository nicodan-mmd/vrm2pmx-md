# Sentry 変換情報収集 方向性

エラー: いままで通り exception を送信

辞書候補: 正常終了時に info レベルのカスタムイベントを送信

実装イメージ

変換終了時に「fallback発生」「skip発生」「推定変換あり」などを判定

条件を満たしたら captureMessage / captureEvent で送信

送るのは構造化JSON（理由コード中心）

モデル固有名はハッシュ化 or 送らない

運用で大事な分離

event_type=error と event_type=dict_candidate を明確にタグ分離

可能ならSentryプロジェクト自体を分ける

alert rule は error のみ対象にする

dict_candidate はダッシュボード集計専用にする

辞書候補イベントの最小項目

vrm_version

fallback_codes（例: RIGIDBODY_OTHERS_FALLBACK, MORPH_BINDS_EMPTY_SKIP）

counts（bones/morphs/rigidbodies/joints）

success=true（正常終了）

app_version, release

注意点

PII除去は必須（ファイルパス、ユーザー名、元モデル名）

高頻度ならサンプリング（例: 10-30%）

タグに高カーディナリティ値を入れない（生モデル名NG）

## 要するに、

「正常終了だけど辞書改善に効く信号」をSentryに送るのは良い運用です。
エラー監視を壊さないために、イベント種別とプロジェクトを分離する設計にすれば安全に回せます。

-- 運用方針 ユーザーの許可

初期値は OFF（未チェック）
文言は明確に
「変換品質改善のため、匿名のモデル情報・変換結果を送信する」
送信内容プレビューを表示
何を送るか事前確認できるようにする
同意があるときだけ送信
いつでも設定変更できる
READMEにも明記（送信項目・送信先・保存期間）
さらに安心感を出すなら、

ファイル名/パスは送らない
ハッシュ化した識別子のみ
エラー監視と品質改善イベントを分離

-- 処理フロー

事前同意チェックボックス（初期OFF）
「品質改善のため匿名エラーレポート送信を許可する」

例外発生
通常のエラーメッセージを表示

同意ONのときだけ確認ダイアログ
「エラーレポート送信フォームを開きますか？」

OK後にフォーム表示
送信内容プレビュー付きで、最終送信ボタンを別に置く

送信成功/失敗を明示
成功時は「ご協力ありがとうございます」、失敗時は再試行導線
