# wasm migration note

## 背景

- 現在の Web PoC は、フロントエンドが Vite + React + TypeScript、バックエンドが FastAPI の前提で整理されている。
- GitHub Pages で公開する場合、サーバーサイド処理は動かないため、変換処理をブラウザ側に寄せる必要がある。
- その候補として、Pyodide による Python 実行と、より本格的な Wasm 化を比較した。

## GitHub Pages 公開について

- 公開対象は `frontend` のビルド成果物である `dist` を使う。
- `public` は Vite のビルド入力用ディレクトリであり、公開物を置く場所ではない。
- 一般的な公開方法は次のいずれか。
  - GitHub Actions でビルドして GitHub Pages にデプロイする。
  - `gh-pages` ブランチのルートに `dist` の中身を配置する。
- リポジトリ名付き URL で公開する場合は、Vite の `base` 設定を合わせる必要がある。
- GitHub Pages は静的ホスティングなので、FastAPI をそのまま動かす構成は使えない。

## Wasm 化の考え方

- 「バックエンド処理を Wasm 化したい」という話は可能ではある。
- ただし、Python バックエンド全体をそのまま Wasm にするより、以下の二択で考えるのが現実的。
  - 変換コアを Rust/C++ などで再実装して Wasm 化する。
  - Python コードを Pyodide 上でブラウザ実行する。
- 工数と初速の観点では、Pyodide のほうが圧倒的に試しやすい。

## 現状構成と Pyodide の相性

- FastAPI 側は比較的薄く、主にファイル受け取り、一時ディレクトリ作成、変換呼び出し、ZIP 返却を担当している。
- 変換本体はサービス層にまとまっており、完全な再実装までは不要そうに見える。
- 一方で、現状はファイルパス前提の処理や GUI 層由来の定数参照が残っており、そのままブラウザ移植はできない。

## Pyodide だけでやる場合の結論

- PoC としてはかなり現実的。
- 「そんなにガチに作らず、まず動かしたい」という目的には合っている。
- ただし、完全ノー変更では無理で、最低限の分離と I/O の置き換えは必要。

## Pyodide で詰まりやすい点

### 1. GUI 依存の分離

- Web/API 側から GUI 層の定数を直接 import している箇所がある。
- こうした定数は純粋な設定モジュールへ切り出したほうがよい。
- 変換コアは `form` 以下に依存しない状態に寄せる必要がある。

### 2. ファイルシステム前提

- 現在は一時ディレクトリ、実ファイル保存、出力ファイル作成、ZIP 化という流れを前提にしている。
- ブラウザでは次のいずれかに寄せる必要がある。
  - Pyodide の仮想ファイルシステムを使う。
  - `bytes` 入力と `bytes` 出力の API に変える。
- 後者のほうが将来的な移植性は高い。

### 3. 依存ライブラリ

- `numpy` は利用しているため、Pyodide 上で動かす前提自体は成立しやすい。
- ただし、依存が重いほど初回ロードと起動時間は悪化する。
- `wxPython` はブラウザでは使えないため、GUI 層は完全に別物として扱う必要がある。

### 4. パフォーマンスと UX

- Pyodide は初回ロードが重い。
- 大きい VRM の処理では、ブラウザメモリ使用量や待ち時間が問題になりやすい。
- 実用化する場合は Web Worker 化や進捗表示がほぼ必須。

## 変更量の目安

- フル Wasm 再実装と比べれば軽い。
- ただし、数ファイルだけ直せば終わる規模ではない。
- 体感としては「小規模ではなく、小〜中規模改修」。

最低限必要な変更は次のとおり。

1. GUI 定数・設定の切り出し
2. `path` ベース I/O の抽象化
3. FastAPI 呼び出し前提から、フロントが Pyodide を直接呼ぶ構成への変更
4. ZIP 出力を Python 側または JavaScript 側で扱う設計への変更
5. 進捗表示、キャンセル、エラーハンドリングの再設計

## おすすめ方針

段階的に進めるなら次の順番が安全。

### Phase 1

- Pyodide 上で「VRM bytes を読み込み、PMX bytes を返す」最小経路だけ作る。
- テクスチャ展開や ZIP 化は後回しにする。

### Phase 2

- React 側から Pyodide を呼び出し、設定入力と結果ダウンロードをつなぐ。
- 重い処理は Web Worker へ移す。

### Phase 3

- `tex` ディレクトリの同梱、ZIP 化、進捗表示、エラー整備まで対応する。
- GitHub Pages 公開を安定運用するならここまで必要。

## 判断メモ

- すぐに GitHub Pages へ載せたいなら、まずはフロントだけを静的公開し、変換は後で Pyodide 化する進め方が安全。
- 「今すぐローカルサーバーをなくしたい」なら、Pyodide は最有力候補。
- 「将来の性能や保守性まで最適化したい」なら、長期的には Python 以外でコア再実装した Wasm のほうが強い。
- 現時点の優先度が低いなら、まずは設計メモとして保持し、着手時に GUI 依存と I/O 依存の棚卸しから始めるのが妥当。

## サーバー常設への懸念

- サーバー通信前提の構成にすると、運用者がサーバー代を継続的に負担する必要がある。
- 個人開発では、実装コストよりも保守運用コストのほうが後から重くなりやすい。
- サーバーが停止した場合、変換機能そのものが使えなくなるため、単一障害点になりやすい。
- 障害対応、監視、セキュリティ更新、ストレージ管理まで含めると、長期運用の負担は無視しにくい。
- そのため、本プロジェクトでは「常設サーバーが前提の構成はできるだけ避けたい」という判断は妥当。

## ブラウザ完結方式の利点

- GitHub Pages のような静的ホスティングでも公開しやすい。
- サーバー代をほぼ不要にできる。
- 自分の管理する API サーバーが止まったことで全体が使えなくなる、というリスクを下げられる。
- 利用者のファイルをサーバーへ送らず、ローカルブラウザ内で処理できる可能性がある。
- 将来的にブラウザ完結に寄せられれば、PWA 化も視野に入る。

## PWA 観点の補足

- ブラウザ内で変換処理が完結する構成に近づくほど、PWA との相性はよくなる。
- PWA 化できれば、インストール可能なアプリのような体験、オフライン利用、更新配布の簡素化といった利点を狙いやすい。
- ただし、重い変換処理を安定して扱うには、Web Worker、キャッシュ戦略、ブラウザストレージ使用量の整理が必要になる。
- したがって、PWA は最初の目標というより、Pyodide などでブラウザ完結の基盤が整った後に検討するのが現実的。
