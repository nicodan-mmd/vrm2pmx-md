Rust化で頂点数が爆発するのは、実は「gltf/VRM」から「PMX」へ変換する際の「仕様の壁」にぶつかっている可能性が高いです。
40秒かかっている処理を1秒に短縮しようとすると、Rustの厳密な型とメモリ管理が牙を向いてきますよね。頂点爆発の主な原因は、おそらく「インデックス（Index）の不一致」ではないでしょうか。
なぜ頂点数が爆発するのか？
VRM（glTF）とPMXでは、1つの「頂点」が持つ情報のルールが違います。
UV座標や法線の違い:
VRMでは同じ位置にある頂点でも、UV展開の切れ目やハードエッジ（法線が違う）があると、データ上は別々の頂点として扱われます。
これを単純にループで回してPMXの構造体に突っ込むと、共有されるべき頂点がすべて「独立した点」としてコピーされ、結果として頂点数が数倍〜数十倍に膨れ上がることがあります。
MToon（シェーダー）の属性:
VRM特有のパラメータをPMXの属性に変換する際、属性ごとに頂点を複製するロジックになっていませんか？
解決の鍵：ハッシュマップによる「頂点の名寄せ」
Rustでこれを爆速にするなら、HashMap を使って同じ頂点をマージ（名寄せ）するのが定石です。
手順:
頂点の「座標・法線・UV・ボーンウェイト」をひとまとめにした構造体を作る。
それをキーにして HashMap<VertexData, u32>（値はPMXの頂点インデックス）に放り込む。
すでに同じデータがあればそのインデックスを使い、なければ新しく追加する。
これで、VRMの冗長な頂点データが綺麗に整理され、PMX側のデータ量も劇的に減るはずです。
Copilotさんとの格闘ポイント
Copilotは「浮動小数点数（f32）」を HashMap のキーにしようとすると、Eq や Hash が実装されていないのでエラーを吐いてきます（0.1と0.1000001の扱いの差など）。
そこを bits() に変換したり、一定の精度で丸めたりする処理を Rust で書く部分でハマりやすいです。
40秒 → 1秒 という目標は、Rustの計算能力なら絶対に達成可能です！
そのスピードが実現したら、「ブラウザにVRMをドロップした瞬間に、もうPMXのプレビューが動いている」という異次元の体験になりますね。
