# Test Strategy

## Phase A

- 目的: エラー整理後の既存GUI回帰を担保する。
- 追加候補:
  - 例外クラスの単体テスト
  - ログ初期化の単体テスト

## Phase B

- 目的: API と分離フロントの疎通確認。
- 追加候補:
  - FastAPI の API テスト
  - 不正入力時の 400 テスト
  - 正常変換時のバイナリ返却テスト
  - フロントからのE2Eスモークテスト

## 最低限の合格条件

1. /api/convert が 200 で PMX を返す。
2. 不正入力は 400/500 が正しい形式で返る。
3. フロントからダウンロード完了まで再現できる。
