# Implementation Plan

## 実装方針

- 一括移植ではなく、機能単位で段階導入する。
- 先に共通化を済ませてから Profile 分離と VRoid 統合を行う。
- 非VRoid互換を守るため、各段階で回帰確認を挟む。

## 実装順序

### Step 1: Detection Layer

1. 判定API（detect_profile）を追加する。
2. 判定理由をログに残せる構造を定義する。
3. 判定不能時の戻り値仕様を固定する。

### Step 2: Common Conversion Layer

1. glTF/VRM読込ユーティリティを抽出する。
2. 画像展開と accessor 読取を抽出する。
3. テクスチャ出力パス生成を抽出する。

### Step 3: GenericProfile 明確化

1. 既存変換経路を GenericProfile に寄せる。
2. 旧来コードとの振る舞い差分を検証する。

### Step 4: VRoidProfile 統合

1. ボーン処理（変換・補正）を導入する。
2. メッシュ/マテリアル処理を導入する。
3. モーフ/表情処理を導入する。
4. 剛体/姿勢補正処理を導入する。

### Step 5: Frontend 最小更新

1. ボタン追加なしで自動判定実行。
2. 判定結果表示とエラー案内の文言調整。
3. 再試行導線の確認。

## 変更単位（小PR戦略）

1. PR-A: 判定層のみ
2. PR-B: 共通化層のみ
3. PR-C: GenericProfile整理のみ
4. PR-D: VRoidProfile（ボーン/メッシュ）
5. PR-E: VRoidProfile（モーフ/物理）
6. PR-F: frontend表示整備
7. PR-G: 回帰テスト追加

## 受け入れゲート

- Gate 1: PR-B完了時に非VRoid回帰合格
- Gate 2: PR-E完了時にVRoidサンプル完走
- Gate 3: PR-G完了時に全サンプル回帰合格

## 事前確認ゲート（壊れそうなら停止）

- Confirm 1: Step 2 完了時に、共通化の差分が想定より広い場合は実装を止めて確認する。
- Confirm 2: Step 4 開始前に、非VRoid回帰で1件でも不安定な兆候があれば実装を止めて確認する。
- Confirm 3: Step 5 で失敗案内文言を変更する前に、判定不能時ポリシーが未確定なら実装を止めて確認する。

## 停止トリガー

- 非VRoid既存成功ケースで PMX生成不能または表示不能が出た場合
- 判定レイヤーの誤判定率が増加した場合
- 影響範囲が src/service と src/mmd をまたいで拡大し、1PRで収まらない場合

## 継続トリガー（表示優先）

- ボーン欠損や一部モーフ欠落があっても、モデル表示が成立する場合は処理を継続する
- 継続時は必ずエラーレポートに「欠損内容」「対象モデル」「再現手順」を記録する

## ロールバック指針

- 各PRは独立ロールバック可能な単位に保つ。
- 回帰NG時は直前PRを戻して原因範囲を限定する。
